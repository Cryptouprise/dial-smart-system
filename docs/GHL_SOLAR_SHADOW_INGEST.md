# HighLevel Solar Exit shadow ingest

This lane accepts signed HighLevel contact events only to create private,
append-only, redacted shadow evidence. It cannot create or update a lead, add a
queue item, call a voice/SMS provider, write back to HighLevel, send a message,
or grant contact/launch authority.

## HighLevel trust contract

The implementation follows HighLevel's current official
[Webhook Integration Guide](https://marketplace.gohighlevel.com/docs/2021-07-28/webhook/WebhookIntegrationGuide/):

- the exact raw request bytes are verified before UTF-8 or JSON parsing;
- `X-GHL-Signature` uses Ed25519 and its official SPKI SHA-256 is pinned to
  `5a2c1b2749d3efcc233fb27d4ab43d56b8d20818c42cd226f6c9a3ff20d3097c`;
- an invalid modern signature never falls back to legacy RSA; and
- although the shared verifier can diagnose legacy `X-WH-Signature` events,
  the Solar endpoint rejects every legacy-only delivery.

HighLevel documents `ContactCreate`, `ContactUpdate`, and `ContactDndUpdate`
separately. Only a fresh, ordered `ContactDndUpdate` can establish the current
HighLevel call-DND observation. Create/update payloads remain
`SUPPRESSION_STATE_UNKNOWN_FOR_EVENT`. National, state, seller-specific,
reassigned-number, ownership, prior-opt-out, wrong-number, complaint, and global
stop evidence remain unknown in this lane, so every receipt remains held or
quarantined even when its consent evidence is complete.

## Disabled-by-default runtime

The endpoint uses the publishable/anon Supabase client, never a service-role
key. Its only database operations are:

1. `get_ghl_shadow_ingest_contract` for one exact enabled HighLevel location;
2. `record_ghl_shadow_ingest_receipt` for one atomic append/replay decision.

Both `SECURITY DEFINER` functions are callable only as `anon` and independently
require a dedicated 256-bit Edge-to-Vault token. All table privileges, the token
assertion helper, and every other helper are revoked from `anon`,
`authenticated`, and `service_role`.

The lane stays unavailable until all of the following exist:

- `GHL_SHADOW_INGEST_RPC_TOKEN` Edge secret;
- its SHA-256 only (not the raw token) in the Vault secret named
  `dial_smart_ghl_shadow_ingest_rpc_token_sha256`;
- `GHL_SHADOW_IDENTIFIER_HMAC_KEY` Edge secret; and
- `GHL_SHADOW_IDENTIFIER_HMAC_KEY_VERSION`, matching the binding's immutable
  `identifier_key_version`; and
- one exact enabled, resolved, hash-bound organization/location/mapping/policy
  row in `ghl_shadow_ingest_bindings`.

Both Edge secrets have the exact form `base64url:<43 unpadded characters>`,
representing 32 CSPRNG-generated bytes. Runtime rejects all-zero, all-same, and
short repeating placeholder material in addition to enforcing the encoding.
Provision secrets through the platform secret manager; do not paste them into
source, migrations, tickets, or command history.

The RPC token is transported inside the encrypted PostgREST request body so the
database can compare its SHA-256 to Vault. Production request-body logging must
remain disabled or redact the `p_rpc_token` property. Application code never
logs it, database exceptions never echo it, and Vault stores only its digest.

Rotate the RPC bearer and identifier HMAC key on a documented schedule and
immediately after any suspected exposure. For the RPC bearer, provision the new
Edge secret and matching Vault digest as one maintenance action; a mismatch
fails closed with non-2xx responses, so HighLevel retries rather than losing an
accepted event. For an identifier-key rotation, create a new binding/mapping
version, set a new immutable `identifier_key_version`, and preserve that version
with every receipt. The Edge runtime fails closed if the configured key version
does not match the enabled binding. Silently changing the key would break
historical HMAC reconciliation. Revoke the old secret only after the new path
has produced a signed owned-fixture receipt.

## Exact mapping and minimization

The binding requires exactly 16 HighLevel custom-field IDs. Display names,
tags, current contact state, current contact source, and type coercions never
substitute for them. In particular:

- `consent_phone` must already be strict NANP E.164 (`+1` plus ten digits with
  valid NPA/NXX leading digits);
- booleans must be JSON booleans, never strings such as `"true"`;
- `property_state` and `calling_state` come from immutable consent fields, not
  the mutable contact state; and
- phone identifiers are HMAC-SHA256 over the immutable `consent_phone` and
  organization scope. The current contact phone is used only for an equality
  check and is never persisted or used as the identifier.

The receipt contains hashes/HMACs, booleans, versions, reason codes, a signed
source timestamp, and decision metadata. It has no raw name, phone, email,
address, tags, custom fields, payload, or notes column. Database checks also
reject common raw-contact keys and raw email/E.164 values inside evidence JSON.

## Replay and ordering

- Exact raw-body retries are unique per installation scope and return `204`
  only after the pre-existing durable receipt is found.
- A signed `webhookId` is tracked independently. Reusing it with a different
  raw-body hash appends a durable `WEBHOOK_ID_PAYLOAD_COLLISION` quarantine.
- Missing `webhookId` is explicit hold evidence; the raw-body hash still gives
  durable retry idempotency.
- Missing, invalid, future, or more-than-15-minute-old source timestamps cannot
  clear DND ordering. The validated timestamp is stored for future event-state
  composition.
- Successful commits, duplicates, holds, and quarantines return privacy-safe
  `204 No Content`. Signature failures and persistence failures return non-2xx
  so HighLevel can retry.
